The method that performs the calculation is calculate_frame_scores, which takes a list of chars (strings)
and calculates the score.

Implementation:
This was a fun and interesting challenge. Having little experience bowling myself, the "aha" moment when I 
figured out how the go-ahead scoring system worked was satisfying. It allowed me to check the logic that was 
generated by Claude Code thoroughly and make a few corrections and adjustments.

On the first pass, the "calculate_frame_scores" generated method was over 70 lines long and difficult to parse. 
It had up to 4 degrees of nested conditional logic in a single function, so I broke the logic for standard 
frames vs the tenth frame into different functions to make it more readable. The initial version also included 
long, redundant comments about the functionality that I condensed for brevity.

The original logic also had some quirks. The pin_value function's spare return statement was the following:
""" 
if roll == '/':
    prev = rolls[idx - 1]
    return 10 - (10 if prev == 'X' else int(prev))
"""
A spare roll should never immediately follow a strike (or another spare, for that matter) since by definition
it involves two rolls that add up to 10 in one frame, so I removed the check. However, it did get me thinking 
about invalid inputs, even if the passed arguments are within the bounds specified by the problem statement
(an array of 1-9, 'X', or '/'). 

Besides the spare-following-a-strike-or-spare check, the input should never have:
1. A frame that starts with a spare
2. A strike as the second roll of a frame
3. A frame total that exceeds 10 pins (not including the bonus roll in the 10th frame)
4. More rolls than is possible in a game

Additionally, the 10th frame has the same logical restrictions applied to a possible 3 roll frame, if the first
roll is a strike or the second is a spare. The main additional validations required were checking if the bonus 
roll was present if it should be absent or vice versa.

I decided to include the validation checks directly in the functions themselves rather than as a separate 
function, partially so it wouldn't require multiple passes through the data, but also in case those functions
ended up getting used by other code that wasn't aware of the separate validation checks. I also decided to throw 
an exception instead of just logging the discrepancy, because if the inputs for a game score are invalid, that 
indicates that something is seriously broken with Alice's pin module and it needs to be addressed immediately.
